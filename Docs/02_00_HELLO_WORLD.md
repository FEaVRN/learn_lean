# Chapter 2: Hello, world!

1)컴파일러, 2)빌드 시스템, 3)패키지 관리자 등 프로그램 작성에 필요한 모든 도구를 갖추고 있습니다.

이 장에서는 프로그래밍 프로젝트를 시작하고, 컴파일하며, 결과를 실행하는 방법을 설명

환경 실행과 상호작용(예: 표준 입력에서 입력을 읽거나 파일을 생성하는 것)을 실행하는 프로그램들은 계산을 수학적 표현식 평가로 이해하는 것과 조화시키기 어려움

이 장은 **세계와 상호작용하는 함수형 프로그램**에 대해 생각할 수 있는 방식을 제공


## 2.1 Running a program

```lean
def main: IO Unit :=
  IO.println "Hello, world!"
```

```sh
lean --run 02_01_Hello.lean
```

### 2.1.1 구조 분석

- `--run` 옵션은 Lean에게 지정된 파일을 컴파일하고, `main` 를 실행하도록 지시
- **명령줄 인자를 받지 않으면 `main` 은 IO Unit 타입이어야 함**
- `main` 은 함수가 아님 (함수였으면 타입 정의 시에 -> 화살표가 있었을 것)
- 수행되어야할 Effect에 대한 설명 (IO Unit)
- Unit은 0bit를 표현하는 값 (내용이 없음)

### 2.1.2 함수형 프로그래밍과 효과

- Lean의 계산 모델은 수학적 표현식의 평가에 기반
- 변수에 정확히 하나의 값만 주어지며, 그 값은 시간이 지나도 바뀌지 않는다
- 같은 표현식을 다시 평가하면 언제나 같은 결과가 나온다. 
- 반면 유용한 프로그램은 세계와 상호작용
- **같은 표현식이 언제나 같은 결과를 돌려주는 언어가, 시간이 지나면서 내용이 바뀔 수 있는 디스크의 파일을 읽는 프로그램을 어떻게 지원할 수 있을까?**


**Lean에서 IO α 는 "아직 실행되지 않은 외부 세계와의 상호작용 계획"**

![Lean & Effects](../Resources/02_01_cafe.png)

- 요리사의 쪽지는 Lean 표현식을 평가해서 만들어지는 IO 액션
- 카운터 직원의 답장은 효과로부터 다시 전달되는 값들을 나타낸다

#### 프로그램의 실행

- 부수 효과 모델은 `Lean 언어`, `컴파일러`, 그리고 `런타임 시스템(RTS)`의 전체 집합이 실제로 작동하는 방식과 꽤 비슷
1. C로 작성된 런타임 시스템의 원시 연산들이 모든 기본 효과를 구현
2. 프로그램이 실행될 때 RTS는 main 액션을 호출
3. 이 액션은 RTS가 실행할 새로운 IO 액션들을 반환
4. RTS는 이 액션들을 실행하고, 계산을 수행하기 위해 사용자의 Lean 코드에 일을 위임
- Lean 내부 관점에서 보면 프로그램에는 부수 효과가 없음
- IO 액션은 그저 수행될 작업에 대한 설명
- 프로그램 사용자라는 외부 관점에서 보면, 프로그램의 핵심 논리에 대한 인터페이스를 만들어 주는 부수 효과의 층이 존재 (RTS)

### 2.1.3 Real-World 함수형 프로그래밍

- Lean에서 부수 효과를 생각하는 또 다른 유용한 방법
- IO 액션을 **“세계 전체를 인자로 받아 값과 새로운 세계를 함께 반환하는 함수”로 보는 것**
- 이 경우 표준 입력에서 한 줄의 텍스트를 읽는 일은 순수 함수다. 매번 서로 다른 세계가 인자로 들어오기 때문이다. 
  - `World1, World2, World3, World4, World5` 
- 표준 출력에 한 줄을 쓰는 일도 순수 함수다. 함수가 돌려주는 세계가 시작할 때의 세계와 다르기 때문. 
  - `World1', World2', World3', World4', World5'`
-  모든 원시 IO 액션이 하나의 세계를 받아 새로운 세계를 돌려주고, 이 불변 조건을 보존하는 도구들로만 그것들을 결합

#### 세계의 토큰화

- 이 모델 자체를 그대로 구현할 수는 없다. 우주 전체를 Lean 값으로 만들어 메모리에 올려둘 수는 없기 때문
- 세계를 대신하는 추상 토큰을 이용한 변형은 구현
1. 프로그램이 시작될 때 이 세계 토큰이 주어지고
2. 이 토큰은 IO 원시 연산으로 전달되며
3. 그들이 돌려준 토큰은 다시 다음 단계로 전달

> 왜 토큰을 전달하나? 핵심은 순서 보장(sequence) 때문이다.
> 
> 부수 효과가 있는 프로그램에서, 표현식의 순서가 중요하다.
>
> 1. println: 당신의 이름은 무엇입니까?
> 2. readline: 김주호
> 3. println: 안녕하세요, 김주호님!

#### **여러 Effect를 조합**하는 방법

- 여러 효과를 사용할 수 있게 하기 위해, Lean에는 **do 표기법**이라는 하위 언어가 있으며
- 이것을 이용해 이런 원시 IO 액션들을 더 크고 유용한 프로그램으로 안전하게 조합

### 2.1.4 IO 액션 합치기

- 대부분의 유용한 프로그램은 출력을 만들 뿐 아니라 입력도 받는다. 
- 더 나아가 입력에 근거해 결정을 내리고, 그 입력 데이터를 계산의 일부로 사용하기도 한다.

```lean
def main: IO Unit := do
  let stdin <- IO.getStdin
  let stdout <- IO.getStdout

  stdout.putStrLn "How would you like to be addressed?"
  let input ← stdin.getLine
  -- :=를 사용하는 이유는 String.dropRightWhile이 IO 액션이 아니라 문자열 위의 일반 함수이기 때문
  let name := input.dropEndWhile Char.isWhitespace 
  stdout.putStrLn s!"Hello, {name}!"
```

```sh
lean --run FpStudy1/02_02_HelloName.lean
```

- main 액션은 do 블록으로 이루어진다. 
- 이 블록에는 문장들의 순서열이 들어 있으며
- 그 문장들은 지역 변수(let으로 도입됨)일 수도 있고 실행될 액션일 수도 있다. 
- **do 문법**은 **명령형 프로그램을 모델링**하는 데 특화된 Lean 내부의 **특수 목적 하위 언어**


#### do 문법

- do 블록 안에서 let은 보통 표현식 안에서의 let과는 약간 다른 의미를 가진다

| 형태           | 실제로 하는 일                                                | 프로그램 관점         |
| -------------- | ------------------------------------------------------------- | --------------------- |
| 일반 let 식    | 바로 뒤따르는 단 하나의 표현식에서만 지역 정의를 사용         | 좁은 스코프           |
| do 블록의 let  | do 블록의 나머지 모든 문장에서 지역 바인딩을 사용             | 블록 전체 스코프      |
| `let x := E`   | E를 평가해 순수 값을 얻고 x에 바인딩                          | 계산                  |
| `let x ← E`    | E를 평가해 IO 액션을 얻고, 그 액션을 실행한 결과를 x에 바인딩 | 효과 실행 + 결과 수집 |
| expr만 있는 줄 | 보통 IO Unit 액션을 만들어 그 줄에서 실행하게 됨              | 효과 수행             |
| do 블록        | 문장을 순서대로 이어 붙여 하나의 큰 IO 액션을 만듦            | 절차적 서술           |

- IO.getStdin과 IO.getStdout이 IO 액션인 이유
- 프로그램 안에서 stdin과 stdout을 지역적으로 덮어쓸 수 있게 하려는 데 있으며, 이는 때때로 편리하다. 만약 이것들이 C처럼 전역 변수였다면 의미 있게 덮어쓸 방법이 없겠지만,
- IO 액션은 매번 실행될 때 서로 다른 값을 돌려줄 수 있다. 

> stdout 자체도 세계 상태(world state)의 일부
>
> Lean에서는 stdout을 현재 실행 환경에서 stdout을 조회하는 IO action
>
> stdout이 고정된 전역값이 아니라 실행 컨텍스트에 따라 달라질 수 있는 값


## 2.2 Step By Step

### 2.2.4 값으로서의 IO Action

- 왜 `표현식 평가`와 `IO 액션 실행`의 구분이 필요한지 잘 보이지 않을 수 있다
- 각 액션은 생성되자마자 바로 실행되는 것처럼 보인다. 그렇다면 다른 언어들처럼 _평가_ 중에 _효과_ 를 곧바로 수행하면 되지 않을까? 

1. 평가와 실행을 분리하면 어떤 함수가 부수 효과를 가질 수 있는지 프로그램이 명시적으로 드러내야 한다.
   - 효과가 없는 부분은 프로그래머의 머릿속에서든 Lean의 형식 증명 도구를 사용하든 수학적으로 훨씬 다루기 쉽기 때문에, 이런 분리는 버그를 줄이는 데 도움이 된다. 
2. 모든 IO 액션이 만들어지자마자 실행되어야 하는 것은 아니다. 액션을 수행하지 않은 채 “말할 수 있다”는 능력 덕분에, 평범한 함수들이 제어 구조로 사용될 수 있다.
   - 액션 결과를 전달한 것이 아니라 액션(실행계획)을 전달한 것이기 때문에 어떻게 액션을 활용할 것인지 결정할 수 있다.

    ```lean
    -- do notation을 활용
    def twice (action : IO Unit) : IO Unit := do
      action
      action

    def nTimes (action : IO Unit) : Nat → IO Unit
      -- pure 함수는 부수 효과는 없지만 인자를 반환하는 IO 액션을 만듬
      | 0 => pure ()
      -- n + 1에서만 do notation으로 여러 액션을 활용
      | n + 1 => do
          action
          nTimes action n
    ```
